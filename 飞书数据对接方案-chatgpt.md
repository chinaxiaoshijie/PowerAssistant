📘 飞书 API 数据结构对接方案
==================

**适用对象：AI + 教育数字化企业（研发优先级最高）**  
**组织规模：≈50 人**

一、总体架构设计
========

1.1 架构目标
--------

* 打通飞书业务数据 → 决策引擎

* 实现结构化指标抽象

* 支持实时/准实时同步

* 保持低耦合，便于扩展

* * *

1.2 总体架构图（逻辑层）
--------------

    飞书 SaaS 层
    │
    ├── 组织架构 API
    ├── 任务 API
    ├── OKR API
    ├── 日历 API
    ├── 多维表格 API
    ├── 文档 API
    │
    ↓
    数据接入层（Connector Layer）
    │
    ├── OAuth2 认证模块
    ├── Webhook 事件监听
    ├── 定时拉取任务
    │
    ↓
    数据标准化层（Normalization）
    │
    ├── 用户实体模型
    ├── 项目实体模型
    ├── 任务实体模型
    ├── 指标抽象模型
    │
    ↓
    决策引擎（AI & Rules）
    │
    ↓
    可视化 & 报告生成

二、飞书 API 对接模块设计
===============

* * *

2.1 应用注册与权限设计
=============

2.1.1 应用类型
----------

推荐使用：

* **企业自建应用**

* 内部应用模式

原因：

* 权限更高

* 无需用户逐个授权

* 适合公司级管理系统

* * *

2.1.2 权限范围建议
------------

| 模块   | 所需权限                         |
| ---- | ---------------------------- |
| 组织架构 | `contact:user.base:readonly` |
| 任务   | `task:task:readonly`         |
| OKR  | `okr:objective:readonly`     |
| 日历   | `calendar:calendar:readonly` |
| 多维表格 | `bitable:app:readonly`       |
| 文档   | `docx:document:readonly`     |
| 消息推送 | `im:message`                 |

三、核心数据模型设计
==========

* * *

3.1 用户实体模型（User）
================

    {
      "user_id": "string",
      "name": "string",
      "department_id": "string",
      "role": "研发/交付/产品/管理",
      "job_level": "P5/P6/...",
      "join_date": "date"
    }

来源：

* `/contact/v3/users`

* `/contact/v3/departments`

用途：

* 计算组织负载

* 单点风险识别

* 研发人力结构分析

* * *

3.2 项目实体模型（Project）
===================

数据来源：

* 多维表格

* 自定义项目表

建议统一结构：
    {
      "project_id": "string",
      "name": "string",
      "type": "研发/交付",
      "owner": "user_id",
      "start_date": "date",
      "end_date": "date",
      "status": "进行中/已完成/延期",
      "priority": "P0/P1/P2",
      "risk_level": "low/medium/high"
    }

* * *

3.3 任务实体模型（Task）
================

来源：

* `/task/v2/tasks`

    {
      "task_id": "string",
      "title": "string",
      "project_id": "string",
      "assignee": "user_id",
      "status": "todo/in_progress/done",
      "priority": "high/medium/low",
      "due_date": "date",
      "completed_at": "date",
      "is_overdue": "boolean",
      "tags": ["技术债", "实验AI", "大模型"]
    }

五、指标抽象层设计（核心）
=============

* * *

5.1 研发健康指数计算结构
==============

    研发健康指数 =
      0.3 * 任务准时率
    + 0.2 * 技术债占比
    + 0.2 * 单点依赖风险
    + 0.3 * 模块完成度

数据来源：

* Task 表

* Project 表

* User 表

* * *

5.2 交付健康指数
==========

    交付健康指数 =
      0.4 * 项目延期率
    + 0.3 * 客户问题反馈率
    + 0.3 * 版本成功率

* * *

六、数据存储设计
========

推荐结构：
6.1 原始数据层（ODS）
--------------

* 原始 JSON 存储

* 保留飞书原字段

6.2 标准化层（DWD）
-------------

统一字段命名：

* user_id

* project_id

* task_id

6.3 指标层（ADS）
------------

只存：

* 日级指标

* 周级指标

* 月级指标
